# 系统总体拓扑结构

## 0. 架构哲学

QuantSaaS 的首要原则是“策略同构”。同一份 `Step()` 纯函数必须同时驱动回测、GA 与实盘，任何差异都只能发生在函数外圈的输入装配与执行收敛层，不能在策略内部出现 `if isBacktest` 之类的分叉。

第二原则是“执行现实优先于回测想象”。v1 固定为 Bitget 现货、`1h` 已完成 bar、纯市价单、保守费用与滑点模型。所有产品设计都要接受这套现实约束，不能为了让曲线更好看而引入无法在真实环境稳定复现的行为。

第三原则是“长期核心资产边界”。本系统不是面向任意山寨币的泛化量化平台，而是面向 `BTCUSDT` 与 `ETHUSDT` 这类长期核心资产的 DCA 增强系统。用户心态、风控语义、Ghost DCA 基准与模板文案都必须围绕这一前提组织。

## 1. 三端物理部署

### 1.1 SaaS（云端）

SaaS 是决策大脑与用户入口。它负责用户认证、模板/实例管理、运行时快照持久化、历史 K 线存储、回测与 GA 协调、以及向 LocalAgent 下发交易意图。SaaS 不保存任何交易所 API Key，也不直接持有用户的交易所会话。

### 1.2 Agent（用户本地）

Agent 是执行手。它只负责登录 SaaS、维持 WebSocket 长连接、执行 Bitget 现货市价单、拉取余额与成交明细，并将 `DeltaReport` 回传云端。Agent 不包含策略代码、不连接数据库、不做自主决策。

### 1.3 Lab（本地算力机）

Lab 是实验室进程，逻辑上与 SaaS 共用代码基座，但在运行角色上只开放历史数据处理、回测与 GA 任务，不开放实盘下发。Lab 与 SaaS 可连接同一套 Postgres/Redis，区别仅体现在 `app_role` 与路由/执行权限。

## 2. app_role 三态行为矩阵

| role | 场景 | 开放能力 | 禁区 |
|---|---|---|---|
| `saas` | 云端产品环境 | 用户认证、模板与实例管理、健康检查、Agent 通信、运行中实例驱动 | 禁止触发 GA 写任务与 Lab 专属路由 |
| `lab` | 本地实验室 | 数据同步、CSV 导入、回测、GA、基因库查看与晋升 | 禁止下发真实交易命令 |
| `dev` | 本地开发 | 所有能力均可开启 | 无生产保障 |

## 3. 模块与职责边界

### 3.1 Strategy

Strategy 模块定义模板元信息、参数结构、运行时状态与 `Step()` 纯函数。首版仅有 `core-btc-v1` 与 `core-eth-v1` 两个模板。两者共享 quant 工具，但拥有独立默认参数、独立冠军缓存 key 与独立基因库。

### 3.2 Instance

Instance 模块负责把“模板 + 标的 + 资金配额 + 用户账号”变成一个可运行实例。它管理实例生命周期、组合快照、待执行命令与策略运行时状态。实例调度只会在 `1h` 已完成 bar 上触发。

### 3.3 Evolution

Evolution 模块是纯计算线程，不触碰实盘账户。它从历史 K 线与冻结的 `SpawnPoint` 出发，搜索更稳健的参数组合，并把结果写为 `challenger`。晋升需要满足硬门槛与人工审核。

### 3.4 Auth

Auth 模块负责用户注册、登录、JWT 鉴权与当前用户信息查询。首版为单租户简化模式，保留用户隔离与静态 plan 字段，但不接真实计费系统。

### 3.5 Data Lab

Data Lab 负责历史 K 线的同步、导入、覆盖范围查看与数据质量反馈。它不是实时行情网关，而是回测、GA、AI 摘要的离线数据入口。

### 3.6 AI Snapshot

AI 模块只生成展示型快照，不进入策略纯函数与 GA。它读取历史 K 线、新闻与情绪摘要，输出一个带来源与时间戳的提示记录，供前端与日志展示。

## 4. 全局状态总线

系统使用单一 Postgres 作为真源数据库，单一 Redis 作为缓存层。Redis 只用于 JWT 会话辅助缓存、冠军参数缓存、AI 摘要缓存等非真源状态，不承担信号传递与订单编排职责。

| 数据 | 真源 | 说明 |
|---|---|---|
| 用户与 plan | Postgres | 单租户简化，但仍按用户隔离实例 |
| 模板与实例 | Postgres | 模板静态注册，实例动态创建 |
| 账户快照与 lot 账本 | Postgres | 所有实盘收敛结果落库 |
| K 线历史数据 | Postgres | 来自 Bitget 同步或 CSV 导入 |
| 冠军/挑战者基因 | Postgres | Redis 只做冠军缓存 |
| Agent API Key | Agent 本地配置 | 绝不上传至 SaaS |
| AI 快照 | Postgres | 用于展示与审计 |

## 5. WebSocket 协议

Agent 通过 `GET /ws/agent` 建立连接，并在第一条消息中发送 `auth`。SaaS 返回 `auth_result` 后，连接进入正常循环。正常消息类型固定为：

- `heartbeat` / `heartbeat_ack`
- `command` / `command_ack`
- `delta_report` / `report_ack`

`TradeCommand` 固定描述纯市价交易意图：

- `client_order_id`
- `action`：`BUY` / `SELL`
- `engine`：`MACRO` / `MICRO`
- `symbol`
- `amount_usdt`
- `qty_asset`
- `lot_type`

`DeltaReport` 固定描述执行后状态：

- `client_order_id`
- `balances`
- `execution`

Agent 离线时，SaaS 记录审计日志并跳过当前可执行命令，不积压待补发旧命令。下一次 `1h` tick 将基于最新快照重新决策。

## 6. 生命周期动作

### 6.1 系统启动

启动顺序固定为：读取配置 -> 初始化 logger -> 连接 DB -> AutoMigrate -> 连接 Redis -> 注册路由 -> 启动 HTTP 服务器。Lab 模式在后续阶段额外启动数据同步与回测/GA 服务。

### 6.2 实盘 tick

同一实例只在最新已完成的 `1h` bar 尚未处理时执行一次 `Step()`。如果 Agent 当前不在线，则只保留本次跳过事件的审计记录，不写待补发命令。

### 6.3 Agent 重连

Agent 断线后以指数退避重连，重新登录并建立 WS 连接。重连成功后首先发送当前余额快照，用于云端恢复组合真相，而不是补发历史订单。

### 6.4 数据导入

Bitget 同步与 CSV 导入都只写 `KLine` 真源表。首版 CSV 只支持 `1h` K 线，使用 `(symbol, interval, open_time)` 做幂等去重。

## 7. 不可推翻的技术决策

1. `Step()` 是唯一策略入口，且必须为纯函数
2. 回测、GA、实盘共用同一 `Step()` 实现
3. 首版仅支持 Bitget 现货 `BTCUSDT` / `ETHUSDT`
4. 首版统一使用 `1h` 已完成 bar 驱动
5. 首版只支持市价单；回测采用保守费用与滑点
6. Agent 离线时不积压旧命令
7. AI 不参与下单、不参与 GA、不写入 `StrategyInput`
8. API Key 仅存在于 `config.agent.yaml`
9. 数据库 schema 真源为 Go struct + AutoMigrate
10. Redis 不是业务真源
