# QuantSaaS v1 实施计划

## 1. 定位

QuantSaaS v1 是一个面向长期核心资产的 DCA 增强型托管系统，不是任意币种通用量化平台，也不是依赖 AI 直接下单的自动交易系统。首版只面向 Bitget 现货 `BTCUSDT` 与 `ETHUSDT`，并将两者视为两个独立策略模板：`core-btc-v1` 与 `core-eth-v1`。

系统核心目标不是通过高频择时击败一切基准，而是在用户愿意长期持有的核心资产上，提供比简单定投更平滑、更有纪律的资金部署与浮动仓位管理体验。产品边界、回测模型、实盘链路与前端交互都围绕这一点展开。

## 2. 开工顺序

1. 冻结真源文档：统一 `系统总体拓扑结构`、`策略数学引擎`、`进化计算引擎` 三份文档。
2. 初始化工程骨架：Go monorepo、最小 SaaS、最小 Agent、最小 React/Vite 前端。
3. 搭建 SaaS 基础设施：config、logger、Postgres、Redis、JWT、基础认证与健康检查路由。
4. 实现 quant 与策略纯函数：数学工具、Ghost DCA、lot 管理、坩埚窗口、市场状态、宏观引擎、微观引擎、`Step()`。
5. 实现回测与 GA：backtest adapter、evolvable strategy、GA engine、epoch service、challenger promote gate。
6. 实现实盘执行：instance manager、cron tick、WebSocket hub、Bitget LocalAgent。
7. 实现前端业务页：auth、dashboard、instances、evolution、backtesting、agents、Data Lab。
8. 实现展示型 AI：collector、scorer、cache、snapshot API 与前端展示卡片。
9. 端到端联调：最小 smoke test、编译、测试、Data Lab 导入与 Bitget 同步链路验证。

## 3. 首刀范围

首个可执行切片只包含以下内容：

- 本实施计划文件
- 三份真源文档冻结
- Go/前端目录骨架
- `config + DB + Redis + JWT + 基础路由`

这一步不实现真实交易、GA、WebSocket、实例业务页，也不实现让 AI 影响订单的能力。交付目标是得到一个能编译的最小 SaaS 进程与一套稳定的项目约束。

## 4. 收紧后的产品边界

- 市场边界：只支持 Bitget 现货 `BTCUSDT` / `ETHUSDT`
- 模板边界：BTC 与 ETH 分别拥有独立模板、冠军参数、基因库
- 时间粒度：实盘、回测、GA 统一使用 `1h` 已完成 bar
- 执行模型：只支持市价单；回测使用 `taker fee + 固定滑点`
- 离线行为：Agent 离线时跳过本次可执行命令，不积压旧指令
- AI 边界：AI 仅做提示与展示，不进入 `Step()`，不进入 GA，不影响订单
- 历史数据：支持 Bitget 历史 K 线同步与 `1h` CSV 导入

## 5. 验收基线

第一刀完成后，应满足以下基线：

- `go test ./...` 可以通过编译型验证
- `go list ./...` 成功
- 最小 SaaS 具备 `/healthz`、`/api/v1/auth/register`、`/api/v1/auth/login`、`/api/v1/auth/me`
- 工程目录结构与文档命名稳定，不再依赖旧文档中的占位符
